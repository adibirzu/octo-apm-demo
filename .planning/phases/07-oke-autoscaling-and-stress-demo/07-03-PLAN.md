<objective>
Ship the in-cluster load engine: a long-lived `octo-stress-runner`
Deployment containing a FastAPI wrapper plus the k6 binary in the same
pod. The wrapper exposes a 4-endpoint internal HTTP API (run, clear, state,
healthz), enforces concurrency=1 with HTTP 409 on race, sends SIGTERM on
clear, and tags every run with a UUID `run_id` that propagates to APM
spans (wrapper) and k6 OTLP output (load). Three k6 scenarios live in
`tools/stress-runner/scenarios/` adapted from the existing `shop/k6/` +
`crm/k6/` scenario libraries.

This plan is the SOLE writer of `deploy/k8s/oke/stress-runner/*`,
`deploy/helm/octo-apm-demo/templates/stress-runner-*`, and
`tools/stress-runner/*`. It does NOT edit `deploy/helm/octo-apm-demo/values.yaml`
— the top-level `stressRunner:` values block is staged by Plan 07-01
(single-owner of values.yaml in Wave 1). Plan 07-03 templates reference that
block read-only.

Purpose: Provides the actual load amplification mechanism the admin
stress-test UI will trigger in plan 05. By splitting the wrapper into its
own plan/pod, the admin route only orchestrates (HTTP POST) — all k6
lifecycle complexity is encapsulated.

Output: Raw OKE manifests under `deploy/k8s/oke/stress-runner/`; Helm
templates under `deploy/helm/octo-apm-demo/templates/`; Python package
under `tools/stress-runner/`; three k6 scenario `.js` files.
</objective>
